## ❌ 부족한 PM 스킬 (개선 필요)

### 🔴 Critical (급한)

#### 12. **MVP 정의 능력** ❌
- **현재:** "Backend 기능 완성 = 완료" 로 생각
- **필요:** "User가 실제로 사용할 수 있어야 완료"
- **예:**
  - 틀림: 스킬 시스템 Backend 완성 → 완료!
  - 옳음: 스킬 시스템 UI 완성 + User가 실제로 스킬 사용 가능 → 완료!
- **개선 방안:**
  - 프로젝트 시작 시 MVP 정의
  - Phase 별로 "User가 할 수 있는 것" 명시

#### 13. **사용자 경험(UX) 중심 사고** ❌
- **현재:** 기능 자체에만 집중
- **필요:** User가 이 기능을 어떻게 쓸지 중요
- **예:**
  - 틀림: 스킬 시스템 → 9종류 스킬, 쿨타임, 버프/디버프
  - 옳음: 스킬 시스템 → AI에게 "선물 주기" 버튼, AI가 "고마워!" 반응
- **개선 방안:**
  - 기능 개발 전에 "User 시나리오" 작성
  - UI에서 User가 실제로 클릭할 것 상상

#### 14. **기획 능력 (왜 이 기능이 필요한지)** ❌
- **현재:** "RPG 니까 스킬 필요" 이런 단순 논리
- **필요:** "User가 스킬 사용 즉, AI가 반응해서 즐거워 하니까 스킬 필요"
- **예:**
  - 틀림: 스킬 → RPG 기능 필요
  - 옳음: 스킬 → AI 상호작용의 다양성 → User 즐거움 ↑
- **개선 방안:**
  - 기획 전에 "Why?" 질문 최소 3번

#### 15. **개발 속도 vs 가치 판단** ❌
- **현재:** 속도 = 16시간에 3Phase 완성, 가치 = 0 (User가 못 씀)
- **필요:** 속도 = 늦더라도, 가치 = User가 즐거운 기능
- **예:**
  - 틀림: Backend 3Phase 16시간, UI 0
  - 옳음: Backend 1Phase, UI 완성 8시간 → User 즉시 사용 가능
- **개선 방안:**
  - "이 기능 지금 필요한가?" 질문
  - "지금 하지 않으면 User가 즐겁지 않다" 확인

#### 16. **사용자 피드백 반영 능력** ❌
- **현재:** 사용자 피드백 없이 개발
- **필요:** 실제 User 피드백 받고 개선
- **예:**
  - 사용자: "이게 재미 있어?" → 나: "User가 실제로 써봐야 알아요!"
  - 사용자: "스킬이 너무 복잡해" → 나: "AI 상호작용 중심으로 단순화"
- **개선 방안:**
  - Phase 별로 User 테스트
  - 피드백을 GitHub Issue에 반영
